# Contributing to ConfigFoundry

Thanks for considering a contribution. The short version:

1. Open an issue first for anything beyond a small fix.
2. Fork, branch, make your change, add tests.
3. Keep `docs/` in sync with the change — documentation drift is
   treated as a real defect here, not an afterthought.
4. Open a pull request describing what changed and why.

The one rule that shapes every review: **never remove an existing
feature or reduce functionality** to make something else cleaner.

For the full guide — reporting bugs, proposing changes, what gets
prioritized in review, codebase-specific gotchas, and how to add a new
permission/storage provider/integration — see
[docs/contributing.md](docs/contributing.md).

Security issues should not be reported as public GitHub issues — see
[SECURITY.md](SECURITY.md).
