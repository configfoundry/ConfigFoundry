3:T3028,<h1 id="api-reference">API Reference<a href="#api-reference" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry exposes a REST API under <code>/api/v1/</code>. This page is a navigable endpoint index with auth/error conventions; the authoritative, always-up-to-date source is the interactive OpenAPI documentation generated from the code itself:</p>
<ul><li><strong>Swagger UI</strong> — <code>http://localhost:8420/docs</code> (try requests directly in the browser)</li><li><strong>ReDoc</strong> — <code>http://localhost:8420/redoc</code> (better for reading/printing)</li><li><strong>Raw OpenAPI JSON</strong> — <code>http://localhost:8420/openapi.json</code></li></ul>
<p>Both docs pages are fully self-hosted (no CDN, works air-gapped) and register a <code>BearerAuth</code> security scheme, so every protected endpoint shows a lock icon and Swagger&#39;s <strong>Authorize</strong> button accepts either a JWT access token or an API key. See <a href="/docs/authentication/">Authentication</a> for how to obtain either, and <a href="/docs/api-versioning/">API Versioning</a> for how new versions get added alongside <code>v1</code> without breaking it.</p>
<h2 id="conventions">Conventions<a href="#conventions" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li>Base path: <code>/api/v1</code></li><li>Auth: <code>Authorization: Bearer &lt;jwt-access-token&gt;</code> or <code>Authorization: Bearer cfk_live_...</code> (API key)</li><li>All endpoints except <code>/auth/login</code>, <code>/auth/mfa/verify</code>, and <code>/auth/refresh</code> require authentication.</li><li>Every protected endpoint additionally requires a specific permission code — see <a href="/docs/rbac/">RBAC</a> for the full catalog and which system roles grant which codes.</li><li>Request/response bodies are JSON (except file exports and Excel imports, which use standard multipart/binary).</li><li>Errors use FastAPI&#39;s standard shape: <code>{&quot;detail&quot;: &quot;...&quot;}</code> for simple errors, or a Pydantic validation error list for <code>422</code> responses.</li><li>Every mutating action (create/update/delete on inventory or security objects) is recorded in the audit log — see <code>GET /api/v1/audit</code>.</li></ul>

<h2 id="pagination-filtering-sorting">Pagination, filtering, sorting<a href="#pagination-filtering-sorting" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>List endpoints (<code>/devices</code>, <code>/bandwidth</code>, <code>/subnets</code>, <code>/users</code>, <code>/audit</code>, etc.) accept standard query parameters — check each endpoint&#39;s schema in <code>/docs</code> for its exact filter fields, since they&#39;re resource-specific (e.g. devices filter by tag/subnet, audit filters by actor/date-range/ resource-type). Where an endpoint supports pagination it uses <code>limit</code>/<code>offset</code> query parameters and returns a <code>total</code> count alongside the page of results.</p>
