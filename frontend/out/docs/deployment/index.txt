3:T1999,<h1 id="deployment">Deployment<a href="#deployment" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>This page covers running ConfigFoundry in production, on a single always-on machine. For deployment with zero internet access, see <a href="/docs/airgap/">Air-Gap Deployment</a> first — the two guides complement each other; this one covers topology and networking, that one covers offline installation specifically.</p>
<h2 id="topology">Topology<a href="#topology" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>ConfigFoundry is a single process (FastAPI + the static frontend export served from the same origin) backed by one database. There is no built-in load balancer, message queue, or worker pool — the intended deployment is one instance per team/environment, run behind a reverse proxy for TLS termination:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">Internet / internal network
        │
        ▼
Reverse proxy (nginx / Caddy / your enterprise LB)  — TLS termination
        │  HTTP, localhost or internal network only
        ▼
ConfigFoundry (python3 server.py / run_offline.sh)  — port 8420 default
        │
        ▼
Database (SQLite file, or PostgreSQL/MySQL/SQL Server — see Storage)</code></pre></div>
<h2 id="reverse-proxy">Reverse proxy<a href="#reverse-proxy" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Terminate TLS at the proxy; ConfigFoundry itself serves plain HTTP. A minimal nginx example:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">nginx</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-nginx">server {
    listen 443 ssl;
    server_name configfoundry.yourcompany.example;

    ssl_certificate     /etc/ssl/certs/configfoundry.crt;
    ssl_certificate_key /etc/ssl/private/configfoundry.key;

    location / {
        proxy_pass http://127.0.0.1:8420;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}</code></pre></div>
<p>Then set <code>CONFIGFOUNDRY_AUTH_TRUSTED_PROXIES</code> to the proxy&#39;s IP/CIDR so <code>X-Forwarded-For</code> is trusted only from that source — see <a href="/docs/configuration/">Configuration</a>. <code>SecurityHeadersMiddleware</code> always sends <code>Strict-Transport-Security</code>; it&#39;s inert over plain HTTP and takes effect once TLS is terminated in front of it.</p>
<h2 id="running-as-a-service">Running as a service<a href="#running-as-a-service" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Use your platform&#39;s standard process supervisor rather than a bare foreground process. <code>systemd</code> example:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">ini</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-ini"># /etc/systemd/system/configfoundry.service
[Unit]
Description=ConfigFoundry
After=network.target

[Service]
Type=simple
User=configfoundry
WorkingDirectory=/opt/configfoundry
Environment=CONFIGFOUNDRY_AUTH_JWT_SECRET=...
Environment=CONFIGFOUNDRY_AUTH_SECRET_ENC_KEY=...
ExecStart=/opt/configfoundry/.venv/bin/python3 server.py --no-browser
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target</code></pre></div>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">sudo systemctl enable --now configfoundry</code></pre></div>
<p>Prefer an environment file (<code>EnvironmentFile=/etc/configfoundry/env</code>) over inline <code>Environment=</code> lines for secrets, with restrictive file permissions (<code>chmod 600</code>).</p>
<h2 id="database-choice-at-this-stage">Database choice at this stage<a href="#database-choice-at-this-stage" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>SQLite is fine for a single-instance deployment of any size a small-to- mid-sized team would produce — it&#39;s the whole database in one file, easy to back up, easy to reason about. Move to PostgreSQL when you need concurrent write throughput beyond what SQLite&#39;s single-writer model comfortably handles, or when your organization&#39;s operational standard requires a managed database server regardless of load. See <a href="/docs/storage/">Storage</a> for how to switch.</p>
<h2 id="zero-downtime-notes">Zero-downtime notes<a href="#zero-downtime-notes" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>There is currently no rolling-restart / blue-green story built in — this is a single-process app, and a restart briefly interrupts service (migrations run automatically and are fast for incremental schema changes). For most deployments — an internal tool used during business hours — a few seconds of downtime during a planned restart is acceptable; if it isn&#39;t for your environment, put a second instance behind the reverse proxy pointed at the same PostgreSQL database and drain traffic manually during upgrades. This isn&#39;t validated end-to-end for concurrent multi-instance writes against SQLite — use PostgreSQL if you go this route.</p>
<h2 id="firewall--network-requirements">Firewall / network requirements<a href="#firewall--network-requirements" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li>Inbound: reverse proxy → ConfigFoundry port (default <code>8420</code>), typically</li></ul>
<p>  localhost-only or restricted to an internal network segment.</p>
<ul><li>Outbound: <strong>none required at runtime</strong> — see <a href="/docs/airgap/">Air-Gap Deployment</a>.</li><li>Database: outbound to your database host/port if using PostgreSQL/MySQL/SQL Server.</li></ul>
