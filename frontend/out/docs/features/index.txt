3:T26cb,<h1 id="features">Features<a href="#features" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>The detailed feature reference for the inventory UI itself. For the enterprise/security/deployment feature set, see <a href="/docs/enterprise/">Enterprise Deployment</a> and <a href="/docs/security/">Security</a> instead — this page covers the day-to-day product surface: inventory management, tags, the Network Tree, and the dashboard.</p>
<h2 id="inventory-management">Inventory management<a href="#inventory-management" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Devices, Bandwidth Capping, and Subnets</strong> — full CRUD for your</li></ul>
<p>  inventory, with both a sortable table view and a card view, instant   client-side search, pagination (10/25/50/100/All rows, with your   preference remembered), and a responsive layout down to mobile. Click   any column header to sort, click again to reverse. All of this runs   entirely in the browser, so it works exactly the same whether the   server is on your LAN or unreachable.</p>
<ul><li><strong>Generate YAML</strong> — one config file per Collector Region, built from</li></ul>
<p>  your current devices and bandwidth caps, with live preview and   download. Devices configured for ICMP or SNMP Trap automatically hide   the SNMPv3 credential form entirely, live, as you change the Config   Type, since they don&#39;t need it. See <a href="/docs/api/#config-generation--history-requires-deploymentexecute--profileread">API Reference § Configuration   generation</a>   for the underlying endpoint.</p>
<ul><li><strong>IP address validation</strong> — both client-side (instant feedback while</li></ul>
<p>  typing) and server-side (so a bad value can&#39;t sneak in through the API   directly). Rows with an invalid IP or CIDR are skipped during import   with a clear count rather than silently corrupting your data.</p>
<ul><li><strong>Excel import/export</strong> — export your current data as an <code>.xlsx</code></li></ul>
<p>  template (including your custom tag columns), edit it offline, and   re-import with a merge or replace mode. Credential column headers are   alias-tolerant (<code>Auth Key</code>, <code>authKey</code>, <code>AuthKey</code> all map to the same   field).</p>
<h2 id="dynamic-tags">Dynamic tags<a href="#dynamic-tags" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Collector Region is the one fixed concept</strong> — mandatory, and it&#39;s</li></ul>
<p>  what Generate YAML groups files by. Everything else — Device Class,   Region, Environment, Country, or anything you invent — is created on   demand through <strong>Manage Tags</strong>, not hardcoded. A tag only exists once   you create it.</p>
<ul><li><strong>One tag, many sections</strong> — a tag can apply to Devices, Bandwidth</li></ul>
<p>  Capping, and Subnets all at once, sharing a single value list across   every section it&#39;s enabled for. Define &quot;Environment&quot; once, use it   everywhere. Tag <em>creation</em> happens on <strong>Manage Tags</strong>; every tag&#39;s   <em>value list</em> (alongside Collector Region&#39;s) lives on <strong>Manage Lists</strong>,   so there&#39;s one place to curate every dropdown&#39;s options.</p>
<ul><li><strong>Tags render as real columns</strong> — each tag shows up as its own column</li></ul>
<p>  in every table (header = tag name, cell = value or empty), not packed   into one generic &quot;Tags&quot; column.</p>
<ul><li><strong>Subnet-based tag inheritance</strong> — tag a subnet once by CIDR instead</li></ul>
<p>  of tagging every device in it individually. Any device whose IP falls   inside that range inherits the tag for any value it doesn&#39;t already   set itself, and the matched subnet is written into the generated YAML   (<code>subnet: 10.1.1.0/24</code>) so it&#39;s traceable from the output alone.</p>
<ul><li><strong>Deleting something in use asks first.</strong> Removing a tag, a tag value,</li></ul>
<p>  or a Collector Region that&#39;s still referenced warns you with the   affected record count before letting you proceed, and deleting a tag   definition never deletes the records that used it, only the tag   reference on them. Same rule applies to the tag delete endpoint — see   <a href="/docs/api/">API Reference</a>.</p>
<h2 id="network-tree">Network Tree<a href="#network-tree" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<blockquote><strong>Known gap, stated plainly:</strong> the Network Tree described below was built for ConfigFoundry&#39;s original vanilla-JS frontend (<code>static/</code>) and has not yet been ported to the current Next.js frontend (<code>frontend/</code>). Whichever frontend is present at startup is served exclusively — see <a href="/docs/architecture/#frontend-architecture">Architecture § Frontend architecture</a> — and the legacy frontend predates the authentication layer (its API client sends no bearer token), so it cannot be exposed as a working fallback alongside the current one. This is tracked in <a href="/docs/roadmap/">Roadmap</a> as a real, open gap, not silently dropped. Everything else on this page is live in the current frontend.</blockquote>
<p>Browsing hundreds of devices as a flat table makes it hard to see how your network is actually laid out. The Network Tree is a spatial diagram instead: Subnets on the left, branching to the Devices inside each one, branching to that device&#39;s Bandwidth Capping rows on the right, built so you can pan and zoom around a few hundred devices without losing your place.</p>
<ul><li><strong>Pan and zoom, Google-Maps style</strong>, with independent scrolling per</li></ul>
<p>  column so a bucket with hundreds of devices doesn&#39;t make the whole   diagram unusably tall.</p>
<ul><li><strong>Click a card to drill in</strong> (subnet → its devices → their bandwidth</li></ul>
<p>  rows); click the already-selected card again for a details panel with   an <strong>Edit</strong> button that opens the same form used everywhere else in   the app. Editing from the diagram updates your data immediately.</p>
<ul><li><strong>Hover to trace a connection</strong> — highlights the connector lines down</li></ul>
<p>  to everything beneath the card you&#39;re hovering, so you can see at a   glance what belongs to what.</p>
<ul><li><strong>Unassigned buckets</strong> for devices with no matching subnet and</li></ul>
<p>  bandwidth rows with no matching device, instead of either disappearing   silently.</p>
<ul><li><strong>Filter with <code>key:value</code> queries</strong> (<code>collector_region:india</code>,</li></ul>
<p>  <code>country:&quot;AWS US&quot;</code> — quote multi-word values) or the dropdowns next to   the filter bar. Filtering narrows what&#39;s shown; it never changes the   diagram&#39;s shape.</p>
<h2 id="dashboard-and-everyday-polish">Dashboard and everyday polish<a href="#dashboard-and-everyday-polish" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Dashboard</strong> — inventory totals with icon-bearing stat cards,</li></ul>
<p>  breakdowns by Collector Region and any custom tag (correctly   accounting for subnet-inherited values, not just directly-stored   ones), generation status, and a recent-activity feed.</p>
<ul><li><strong>Dark and light mode</strong> — toggle in the top bar, remembered in your</li></ul>
<p>  browser, applied before the page even paints so there&#39;s no flash of   the wrong theme. The same toggle and preference apply to the   <a href="/docs/">in-app documentation viewer</a> you&#39;re reading this in.</p>
<ul><li><strong>Audit log + YAML history</strong> — every change is attributed to whoever</li></ul>
<p>  made it, and every generation is saved so you can look back at what   was produced and when. See <a href="/docs/api/">API Reference</a> and   <a href="/docs/rbac/#permission-catalog">RBAC</a> for <code>audit:read</code>.</p>
<ul><li><strong>Minimal, well-known dependencies.</strong> The backend runs on FastAPI,</li></ul>
<p>  SQLAlchemy 2.x, Pydantic v2, and uvicorn — all vendored for fully   offline installs (see <a href="/docs/airgap/">Air-Gap Deployment</a>). AES-256-GCM   credential encryption and the YAML serializer are pure-Python with no   additional dependencies.</p>
<ul><li><strong>Safe upgrades.</strong> Every schema change ships as a versioned, idempotent</li></ul>
<p>  Alembic migration that runs automatically on server startup — see   <a href="/docs/upgrade/">Upgrade Guide</a> and <a href="/docs/migrations/">Migrations</a>.</p>
<h2 id="what-this-deliberately-isnt">What this deliberately isn&#39;t<a href="#what-this-deliberately-isnt" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Not a CMDB.</strong> It tracks the fields needed to generate monitoring</li></ul>
<p>  config (IP, region, credentials, a handful of tags), not ownership,   lifecycle, warranty, or asset relationships beyond   subnet/device/bandwidth.</p>
<ul><li><strong>Single-tenant inventory, multi-tenant auth.</strong> The RBAC/API-key/policy</li></ul>
<p>  layer is fully organization-scoped, but the inventory tables   themselves aren&#39;t yet split per-organization — see   <a href="/docs/enterprise/#multi-tenancy">Enterprise § Multi-tenancy</a>.</p>
<ul><li><strong>Single shared database, ordinary write semantics.</strong> With SQLite (the</li></ul>
<p>  default), two people editing the same record at the same moment is   last-write-wins — there&#39;s no optimistic-locking or conflict warning.   For the team-sized usage this was built for, that hasn&#39;t been a   problem in practice, but it isn&#39;t battle-tested under heavy concurrent   write load; move to PostgreSQL (see <a href="/docs/storage/">Storage</a>) if that   matters for your deployment.</p>5:["slug","features","d"]
