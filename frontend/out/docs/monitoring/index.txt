<h2 id="liveness--readiness">Liveness / readiness<a href="#liveness--readiness" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>ConfigFoundry does not currently expose a dedicated <code>/health</code> or <code>/healthz</code> endpoint — this is a known gap, tracked in <a href="/docs/roadmap/">Roadmap</a>, not a silent omission. Until it lands, use one of these as a practical substitute:</p>
<ul><li><strong><code>GET /openapi.json</code></strong> — cheapest option. No authentication required,</li></ul>
<p>  returns quickly, and only succeeds if the FastAPI app is fully booted   (route registration completed). Good enough for a process supervisor   or load balancer liveness probe.</p>
<ul><li><strong><code>GET /api/v1/meta</code></strong> (requires <code>meta:read</code>) — a stronger readiness</li></ul>
<p>  signal, since it exercises the full stack: auth, the service layer,   and a real database round-trip via <code>StorageProvider</code>.</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">curl -sf http://localhost:8420/openapi.json &gt; /dev/null &amp;&amp; echo &quot;up&quot; || echo &quot;down&quot;</code></pre></div>
<p>Internally, every <code>StorageProvider</code> implements a non-raising <code>health_check()</code> returning <code>HEALTHY</code>/<code>DEGRADED</code>/<code>UNHEALTHY</code> with a latency measurement (see <a href="/docs/storage-architecture/#storageprovider-interface">Storage Architecture</a>) — this is the building block a future <code>/health</code> endpoint will expose directly; it isn&#39;t wired to an HTTP route yet.</p>
<h2 id="logs">Logs<a href="#logs" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Structured logging is built in — see <a href="/docs/logging/">Logging</a> for the full reference. For log-based monitoring/alerting, set:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">export CONFIGFOUNDRY_LOG_JSON=true
export CONFIGFOUNDRY_LOG_FILE=/var/log/configfoundry/app.log</code></pre></div>
<p>and point your log shipper (Fluent Bit, Filebeat, Vector, or your platform&#39;s standard agent) at that file. Every request is tagged with a correlation ID (<code>CorrelationIDMiddleware</code>), so a single request&#39;s log lines can be traced end-to-end even under concurrent load.</p>
<h2 id="what-to-alert-on">What to alert on<a href="#what-to-alert-on" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Since there&#39;s no built-in metrics exporter yet (see Roadmap), monitor at the infrastructure and log layer:</p>
<ul><li><strong>Process health</strong>: is the systemd unit / supervisor process running</li></ul>
<p>  and not crash-looping (<code>systemctl status configfoundry</code> /   <code>journalctl -u configfoundry -f</code>).</p>
<ul><li><strong>HTTP availability</strong>: the <code>/openapi.json</code> check above, polled by your</li></ul>
<p>  external monitoring on a short interval.</p>
<ul><li><strong>Database health</strong>: for PostgreSQL/MySQL/SQL Server, monitor the</li></ul>
<p>  database server itself with your standard tooling; for SQLite,   monitor disk space on the volume holding the <code>.db</code> file (SQLite fails   hard, not gracefully degraded, when the disk fills).</p>
<ul><li><strong>Failed login rate</strong>: spikes in <code>401</code>/<code>429</code> responses on</li></ul>
<p>  <code>/api/v1/auth/login</code>, visible in the request log or the audit log   (<code>GET /api/v1/audit/search</code>) — a signal for credential stuffing /   brute-force attempts, distinct from the automatic account lockout   (<code>CONFIGFOUNDRY_AUTH_LOCKOUT_THRESHOLD</code>) which handles individual   accounts but won&#39;t itself alert you.</p>
<ul><li><strong>Certificate expiry</strong>: if TLS is terminated at your reverse proxy</li></ul>
<p>  (the standard setup — see <a href="/docs/deployment/">Deployment</a>), monitor   certificate expiry at that layer, not inside ConfigFoundry.</p>
<h2 id="audit-log-as-an-operational-signal">Audit log as an operational signal<a href="#audit-log-as-an-operational-signal" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Beyond compliance, <code>GET /api/v1/audit/search</code> (requires <code>audit:read</code>) is a practical day-to-day tool: filter by actor, date range, resource type, or result to investigate &quot;who changed this bandwidth cap&quot; or &quot;why did this config generation fail&quot; without needing separate application metrics. See <a href="/docs/rbac/">RBAC</a> for which role (Auditor, or any role with <code>audit:read</code> granted) can query it.</p>
