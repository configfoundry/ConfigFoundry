<h2 id="principles">Principles<a href="#principles" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>These have governed every design decision in ConfigFoundry, including the security, storage, and air-gap work layered on top of the original inventory core:</p>
<ol><li>Core owns the inventory model.</li><li>Integrations are optional.</li><li>Core never imports integrations.</li><li>Zero required backend dependencies beyond what&#39;s vendored.</li><li>SQLite is the default storage; other databases are opt-in, never required.</li><li>Everything must work offline (see <a href="/docs/airgap/">Air-Gap Deployment</a>).</li><li>Explicit code is preferred over clever abstractions.</li><li>Simplicity is a feature — a smaller surface area is easier to audit,</li></ol>
<p>   which matters in the regulated environments this targets.</p>
<p>For where these principles are headed next, see <a href="/docs/roadmap/">Roadmap</a>.</p>

<h2 id="request-lifecycle">Request lifecycle<a href="#request-lifecycle" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ol><li>Reverse proxy terminates TLS, forwards to ConfigFoundry.</li><li><code>TrustedProxyMiddleware</code> resolves the real client IP from</li></ol>
<p>   <code>X-Forwarded-For</code>, trusting it only from configured proxy CIDRs    (<code>CONFIGFOUNDRY_AUTH_TRUSTED_PROXIES</code>).</p>
<ol><li><code>AccessPolicyMiddleware</code> evaluates IP allow/deny rules — runs before</li></ol>
<p>   authentication, since a denied IP shouldn&#39;t even reach the login    endpoint.</p>
<ol><li><code>RateLimitMiddleware</code> throttles per-IP, stricter on <code>/auth/login</code>.</li><li><code>SecurityHeadersMiddleware</code> sets CSP, HSTS, X-Frame-Options, etc. on</li></ol>
<p>   every response.</p>
<ol><li>The route handler runs. Protected routes depend on</li></ol>
<p>   <code>get_current_principal()</code> (resolves a JWT or API key into a    <code>Principal</code>, via <code>Security(HTTPBearer(...))</code> so it&#39;s visible in the    OpenAPI schema) and <code>require_permission(&quot;resource:action&quot;)</code> (checks    the principal&#39;s role grants — see <a href="/docs/rbac/">RBAC</a>).</p>
<ol><li>The handler calls into a service, which calls a repository, which</li></ol>
<p>   goes through <code>StorageProvider.get_engine()</code>.</p>
<ol><li>Security-sensitive and business-mutating actions are recorded via</li></ol>
<p>   <code>AuditRepository.log(...)</code>.</p>
<p>Full detail on this pipeline, including exact middleware registration order and why it&#39;s the reverse of execution order, is in <a href="/docs/authentication/">Authentication</a>.</p>
<h2 id="layering-rules">Layering rules<a href="#layering-rules" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Repositories</strong> never contain business logic — only persistence.</li><li><strong>Services</strong> never import a database driver directly — only repository</li></ul>
<p>  interfaces (ABCs), so swapping SQLite for PostgreSQL touches zero   service code.</p>
<ul><li><strong>Routes</strong> never contain business logic — they validate input,</li></ul>
<p>  call a service, and shape the response.</p>
<ul><li><strong>Core never imports integrations</strong> — the inventory/generation core</li></ul>
<p>  has no knowledge of any specific external system; integrations (if   added later) depend on core, never the reverse.</p>

<ul><li><strong><a href="/docs/authentication/">Authentication</a></strong> / <strong><a href="/docs/authorization/">Authorization</a></strong></li></ul>
<p>  / <strong><a href="/docs/rbac/">RBAC</a></strong> — the full security layer: JWT + refresh   rotation, MFA, API keys, permission-code-based access control, the   Access Policy Engine.</p>

<h2 id="frontend-architecture">Frontend architecture<a href="#frontend-architecture" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Next.js 14 (App Router), built as a fully static export (<code>output: &#39;export&#39;</code>) — no Node.js server process at runtime, no server-side rendering. The static export is served directly by FastAPI&#39;s <code>StaticFiles</code> mount, so the frontend and API share one origin, one port, and one TLS certificate. This also means the CSP can be same-origin only (<code>default-src &#39;self&#39;</code>) with no CDN allowlisting required — see <a href="/docs/security/">Security</a>.</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">frontend/
  src/
    app/            App Router pages (route = folder)
    components/      shared UI components
    lib/              API client, auth context, theme
  out/               built static export (served by FastAPI)</code></pre></div>
<p>The one deliberate exception to <code>&#39;unsafe-inline&#39;</code>-free CSP is <code>script-src</code>: Next.js&#39;s App Router embeds hydration data in an inline <code>&lt;script&gt;</code> tag, which requires <code>&#39;unsafe-inline&#39;</code> regardless of static export mode. All other directives remain strictly same-origin.</p>
<h2 id="configuration-generation-pipeline">Configuration generation pipeline<a href="#configuration-generation-pipeline" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><code>POST /api/v1/generate</code> reads the current inventory (devices, bandwidth caps, subnets, tags) through the service layer, applies the collector YAML template, and writes a <code>history</code> entry recording the generated output alongside who generated it and when — so config drift is always attributable and reviewable via <code>GET /api/v1/history</code>.</p>
