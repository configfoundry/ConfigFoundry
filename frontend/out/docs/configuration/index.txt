3:T2194,<h1 id="configuration-reference">Configuration Reference<a href="#configuration-reference" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry is configured through three layers, in increasing priority:</p>
<ol><li><strong>Built-in defaults</strong> — safe enough to run, not necessarily what you</li></ol>
<p>   want in production (e.g. an auto-generated JWT secret that changes on    every restart, which invalidates all sessions — fine for a laptop demo,    wrong for anything long-lived).</p>
<ol><li><strong>A YAML config file</strong>, passed via <code>--config</code>.</li><li><strong>Environment variables</strong>, which override both.</li></ol>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">python3 server.py --config /etc/configfoundry/config.yaml
python3 server.py --db /path/to/configfoundry.db   # shortcut for SQLite-only</code></pre></div>

<h2 id="authentication--security-configfoundry_auth_">Authentication &amp; security (<code>CONFIGFOUNDRY_AUTH_*</code>)<a href="#authentication--security-configfoundry_auth_" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>All of these come from <code>core/security/config.py::SecurityConfig</code>. Every one has a safe default; the ones marked <strong>set this in production</strong> are the only ones you need to think about before going live.</p>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Env var</th><th>Default</th><th>Description</th></tr></thead><tbody><tr><td><code>CONFIGFOUNDRY_AUTH_JWT_SECRET</code></td><td>random, regenerated each boot</td><td>HMAC signing secret for access tokens. <strong>Set this in production</strong> — an unset secret means every restart invalidates every session.</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_SECRET_ENC_KEY</code></td><td>random, regenerated each boot</td><td>Base64-encoded 32-byte AES-256-GCM key encrypting MFA seeds at rest. <strong>Set this in production</strong> — an unset key means every restart makes existing users&#39; MFA enrollment undecryptable, forcing re-enrollment.</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_ACCESS_TTL_MIN</code></td><td><code>15</code></td><td>Access token lifetime, minutes</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_REFRESH_TTL_DAYS</code></td><td><code>14</code></td><td>Refresh token lifetime, days</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_ARGON2_TIME_COST</code></td><td><code>3</code></td><td>Argon2id time cost</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_ARGON2_MEMORY_COST_KB</code></td><td><code>65536</code> (64 MiB)</td><td>Argon2id memory cost</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_ARGON2_PARALLELISM</code></td><td><code>4</code></td><td>Argon2id parallelism</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_PASSWORD_MIN_LENGTH</code></td><td><code>12</code></td><td>Minimum password length</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_LOCKOUT_THRESHOLD</code></td><td><code>5</code></td><td>Failed logins before account lockout</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_LOCKOUT_MINUTES</code></td><td><code>15</code></td><td>Lockout duration</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_MFA_REQUIRED_ROLES</code></td><td>unset</td><td>Comma-separated role names that must enroll MFA</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_SESSION_IDLE_MINUTES</code></td><td><code>30</code></td><td>Idle session timeout</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_RATE_LIMIT_LOGIN</code></td><td><code>10/60</code></td><td><code>&lt;count&gt;/&lt;window-seconds&gt;</code> for <code>/auth/login</code> and <code>/auth/mfa/*</code></td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_TRUSTED_PROXIES</code></td><td>unset</td><td>Comma-separated CIDRs allowed to set <code>X-Forwarded-For</code> — see <a href="/docs/deployment/">Deployment</a></td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_CORS_ORIGINS</code></td><td>unset (no cross-origin access)</td><td>Comma-separated allowed migration)</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_BOOTSTRAP_PASSWORD</code></td><td>randomly generated, printed once</td><td>Initial Super Admin password</td></tr></tbody></table></div>
<p><strong>Set this in production</strong>, at minimum:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">export CONFIGFOUNDRY_AUTH_JWT_SECRET=&quot;$(openssl rand -base64 48)&quot;
export CONFIGFOUNDRY_AUTH_SECRET_ENC_KEY=&quot;$(openssl rand -base64 32)&quot;
export CONFIGFOUNDRY_AUTH_BOOTSTRAP_EMAIL=&quot;admin@yourcompany.example&quot;
export CONFIGFOUNDRY_AUTH_BOOTSTRAP_PASSWORD=&quot;$(openssl rand -base64 24)&quot;
export CONFIGFOUNDRY_AUTH_CORS_ORIGINS=&quot;https://configfoundry.yourcompany.example&quot;</code></pre></div>
<p>Generate both secrets once, store them in your secrets manager, and reuse the same values on every restart — rotating <code>JWT_SECRET</code> deliberately logs everyone out (sometimes exactly what you want, e.g. after a suspected compromise); rotating <code>SECRET_ENC_KEY</code> makes existing MFA enrollments undecryptable, so treat it as a one-way door.</p>
