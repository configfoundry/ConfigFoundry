3:T1a96,<h1 id="soc-2-control-mapping">SOC 2 Control Mapping<a href="#soc-2-control-mapping" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>This maps ConfigFoundry&#39;s authentication/RBAC/policy layer to the SOC 2 Security (&quot;Common Criteria&quot;) Trust Services Criteria most relevant to an access-control layer. It is a mapping to <em>this codebase</em>, not a certification -- a SOC 2 audit also covers organizational controls (background checks, vendor management, physical security, change management process, etc.) that no application code can satisfy on its own. Treat this as the evidence index an auditor would ask for regarding &quot;how does the application enforce access control and log security events.&quot;</p>
<p>The architecture is intentionally modular (Access Policy Engine, Authentication, and Authorization are separate layers -- see <a href="/docs/authentication/"><code>authentication.md</code></a>) so <strong>GDPR-specific controls can be layered on later</strong> (data export/erasure endpoints, consent tracking) without touching the authentication or RBAC internals. Not built in this pass; the audit trail and per-user data model already support it.</p>

<h2 id="explicitly-out-of-scope-for-this-pass">Explicitly out of scope for this pass<a href="#explicitly-out-of-scope-for-this-pass" class="heading-anchor" aria-label="Link to this section">#</a></h2>
