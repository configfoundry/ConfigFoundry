3:T18c8,<h1 id="authorization">Authorization<a href="#authorization" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>This page covers how ConfigFoundry decides whether an authenticated caller is allowed to do something. For how a caller becomes authenticated in the first place, see <a href="/docs/authentication/">Authentication</a>. For the full permission catalog and system role definitions, see <a href="/docs/rbac/">RBAC</a>.</p>
<h2 id="model">Model<a href="#model" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Authorization is entirely permission-code based — <code>&lt;resource&gt;:&lt;action&gt;</code> strings such as <code>inventory:write</code> or <code>user:manage</code>, defined once in <code>core/security/permissions.py</code>. Route handlers never check a role name directly:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">@router.delete(&quot;/devices/{device_id}&quot;)
def delete_device(
    device_id: int,
    principal: Principal = Depends(require_permission(&quot;inventory:write&quot;)),
):
    ...</code></pre></div>
<p><code>require_permission(code)</code> is a FastAPI dependency that:</p>
<ol><li>Resolves the caller&#39;s <code>Principal</code> via <code>get_current_principal()</code> (JWT</li></ol>
<p>   or API key — see Authentication).</p>
<ol><li>Loads the principal&#39;s effective permission set (union of all</li></ol>
<p>   permissions granted by all roles assigned to the user, or the    explicit permission list attached to an API key if scoped).</p>
<ol><li>Returns the <code>Principal</code> if the code is present; raises <code>403</code> if not.</li></ol>
<p>Because handlers depend on a <em>code</em>, not a <em>role name</em>, what roles grant what permissions is entirely data-driven — changing a role&#39;s grants through the admin UI (<code>PATCH /api/v1/roles/{id}/permissions</code>) takes effect immediately for every route that checks that code, with zero code changes.</p>
<h2 id="newly-created-users-start-with-nothing">Newly created users start with nothing<a href="#newly-created-users-start-with-nothing" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>There is no implicit access &quot;just because an account exists.&quot; A new user holds zero permissions until an admin explicitly assigns a role via <code>POST /api/v1/users/{id}/roles</code>. This is deliberate: it means access review is always a positive, auditable action rather than something to notice is <em>missing</em>.</p>
<h2 id="immediate-revocation-without-a-token-blacklist">Immediate revocation without a token blacklist<a href="#immediate-revocation-without-a-token-blacklist" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Every user row carries a <code>perm_version</code> integer, embedded in every access token issued to that user. Role changes, forced logout, password changes, and account deactivation all increment it. <code>get_current_principal()</code> rejects any token whose embedded <code>perm_version</code> doesn&#39;t match the current database value — so revoking a role, or deactivating an account, takes effect on the very next request, not &quot;whenever the 15-minute access token happens to expire.&quot; This avoids needing a persistent, ever-growing token blacklist while still getting immediate revocation.</p>
<h2 id="organization-scoping">Organization scoping<a href="#organization-scoping" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The entire security layer (<code>User</code>, <code>Role</code>, <code>ApiKey</code>, <code>NetworkACL</code>, <code>AuditLog</code>) is scoped to an <code>Organization</code> and ready for multiple tenants. Every permission check is implicitly scoped to the caller&#39;s own organization — a Super Admin is the one exception, holding every permission across every organization by definition.</p>
<p>See <a href="/docs/authentication/#known-scope-boundaries">Authentication § Known scope boundaries</a> for what is and isn&#39;t retrofitted with org-scoping on the pre-existing inventory tables.</p>
<h2 id="the-access-policy-engine-ip-based-authorization">The Access Policy Engine (IP-based authorization)<a href="#the-access-policy-engine-ip-based-authorization" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>A second, independent authorization layer runs <em>before</em> authentication: <code>AccessPolicyMiddleware</code> evaluates configured IP allow/deny rules (<code>NetworkACL</code> records, managed via <code>/api/v1/policies/network-acls</code>) against the resolved client IP. This catches disallowed traffic earlier in the pipeline than permission checks — a request from a denied IP never reaches the login endpoint, let alone a protected route.</p>
<p>Rules are evaluated first-match-wins, ordered by <code>priority</code> (lower evaluated first). If nothing matches: default is permissive, <em>unless</em> at least one <code>allow</code> rule exists for that scope, in which case the presence of an allow rule signals &quot;allowlist mode&quot; and anything unmatched is denied. Both IPv4 and IPv6 are supported using the standard library&#39;s <code>ipaddress</code> module — no extra dependency.</p>
<p><strong>Known limitation, stated plainly:</strong> time-based access windows (maintenance windows / business-hours-only access) are scaffolded (<code>PolicyEngine.evaluate_time_window()</code>) but always return &quot;allowed&quot; — not yet enforced. Flagged here rather than silently missing.</p>
<h2 id="auditing-every-decision">Auditing every decision<a href="#auditing-every-decision" class="heading-anchor" aria-label="Link to this section">#</a></h2>
