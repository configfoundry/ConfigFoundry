3:T2d89,<h1 id="air-gap-deployment-guide">Air-Gap Deployment Guide<a href="#air-gap-deployment-guide" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry is built to run in environments with <strong>zero internet access</strong> — banks, government, defense, healthcare, telecom, and any Fortune 500 network segment where the machine running ConfigFoundry cannot and will not reach the public internet. This page explains why that constraint shaped the architecture, exactly what &quot;zero internet access&quot; is verified to mean, how to install and update under it, and how to regenerate the offline bundles when you need to bump a dependency.</p>
<h2 id="why-air-gap-support-exists">Why air-gap support exists<a href="#why-air-gap-support-exists" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Most web frameworks assume <code>pip install</code> and <code>npm install</code> can always reach a package registry, and many quietly assume a CDN is reachable too (Google Fonts, jsDelivr, unpkg, Bootstrap CDN — FastAPI&#39;s own default <code>/docs</code> page is CDN-backed out of the box). That assumption breaks completely inside a network with no route to the internet, which is the normal operating environment for a lot of the infrastructure ConfigFoundry is meant to sit in front of. Rather than bolt on an offline mode later, every dependency here is vendored and pinned, and every static asset is self-hosted.</p>
<h2 id="what-air-gapped-is-verified-to-mean">What &quot;air-gapped&quot; is verified to mean<a href="#what-air-gapped-is-verified-to-mean" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Concretely, and checked automatically (not just asserted):</p>
<ol><li>**No PyPI, npm registry, GitHub releases, jsDelivr, unpkg, Google</li></ol>
<p>   Fonts, CDNJS, Cloudflare CDN, Microsoft CDN, Bootstrap CDN, Font    Awesome CDN, or any other external HTTP endpoint** is contacted    during install, build, or runtime.</p>
<ol><li>Every Python dependency is vendored as a pinned wheel in</li></ol>
<p>   <code>vendor/python/</code> with a SHA-256 checksum manifest.</p>
<ol><li>Every frontend JS/CSS/font/icon asset is bundled into the Next.js</li></ol>
<p>   static export or self-hosted under <code>static/vendor/</code> — none are    fetched from a CDN at runtime (this is also what fixed the earlier    CSP/blank-page issues — see <a href="/docs/security/">Security</a>).</p>
<ol><li><code>scripts/validate_airgap.py</code> scans the entire source tree for</li></ol>
<p>   un-allowlisted <code>http://</code>/<code>https://</code> references, confirms    <code>requirements.txt</code>/<code>requirements-dev.txt</code>/<code>frontend/package.json</code> are    pinned to exact versions (no <code>^</code>, <code>~</code>, <code>&gt;=</code>, <code>latest</code>), verifies the    wheelhouse checksums, and — the check that can&#39;t be faked — actually    runs <code>pip install --no-index --find-links vendor/python</code> into a    throwaway virtualenv.</p>
<ol><li>CI&#39;s <code>offline-install-smoke-test</code> job goes one step further:</li></ol>
<p>   it firewalls the runner off from <code>pypi.org</code> and    <code>files.pythonhosted.org</code> with <code>iptables</code> <em>before</em> running    <code>install_offline.sh</code>, so a passing run is proof the install path is    structurally incapable of reaching those hosts, not just that it    didn&#39;t happen to try.</p>
<h2 id="installing-on-an-air-gapped-machine">Installing on an air-gapped machine<a href="#installing-on-an-air-gapped-machine" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">./install_offline.sh      # Linux/macOS
.\install_offline.ps1     # Windows PowerShell</code></pre></div>
<p>What it does, step by step:</p>
<ol><li>Verifies Python 3.10+ is on <code>PATH</code>.</li><li>Confirms <code>vendor/python/</code> exists and has wheels (fails fast with a</li></ol>
<p>   clear message if you&#39;re on a copy of the repo that&#39;s missing them —    this isn&#39;t a valid offline bundle without them).</p>
<ol><li>Creates <code>.venv/</code> and runs</li></ol>
<p>   <code>pip install --no-index --find-links vendor/python -r requirements.txt</code>.    <code>--no-index</code> isn&#39;t just a preference here — it makes pip structurally    unable to reach PyPI even if the network happened to be open.</p>
<ol><li>Uses the pre-built <code>frontend/out/</code> static export if present (the</li></ol>
<p>   normal case — it ships in the release bundle, so <strong>Node.js is not    needed at all</strong> for a standard install). If it&#39;s missing and Node is    available, it attempts an offline rebuild using    <code>vendor/npm/node_modules.tar.gz</code> and the platform-matched native    compiler under <code>vendor/npm/swc-binaries/</code>.</p>
<ol><li>Prints a friendly success message with the next command to run.</li></ol>
<p>Then start it:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">./run_offline.sh          # or .\run_offline.ps1</code></pre></div>
<p>Database migrations run automatically on first launch — there&#39;s no separate offline migration step.</p>
<h2 id="updating-an-existing-air-gapped-install">Updating an existing air-gapped install<a href="#updating-an-existing-air-gapped-install" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">./upgrade_offline.sh --db-from /path/to/current/configfoundry.db</code></pre></div>
<p>This backs up the existing database (timestamped, <code>db/configfoundry.db.pre-upgrade-&lt;timestamp&gt;.bak</code>), copies it forward, re-runs <code>install_offline.sh</code> against the new release&#39;s vendored dependencies, and reminds you that migrations apply automatically on next start. See <a href="/docs/upgrade/">Upgrade Guide</a> for the general upgrade process (version compatibility, rollback) and <a href="/docs/release-process/">Release Process</a> for what&#39;s inside a release bundle.</p>
<h2 id="targeting-a-different-platform">Targeting a different platform<a href="#targeting-a-different-platform" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><code>vendor/python/</code> contains wheels for multiple manylinux tags per architecture (<code>manylinux2014</code>, <code>manylinux_2_28</code>, <code>manylinux_2_34</code> for both <code>x86_64</code> and <code>aarch64</code>) because different upstream packages publish under different tags. If <code>install_offline.sh</code> fails to find a compatible wheel for your exact machine, run:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">python3 -m pip debug --verbose   # lists this machine&#39;s compatible tags</code></pre></div>
<p>and compare against the filenames in <code>vendor/python/CHECKSUMS.sha256</code>. If your platform genuinely isn&#39;t covered, regenerate the wheelhouse (see below) on a machine with internet access and matching target architecture, then copy <code>vendor/python/</code> over.</p>
<h2 id="regenerating-the-offline-bundles">Regenerating the offline bundles<a href="#regenerating-the-offline-bundles" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>You need internet access for this step — it&#39;s meant to run in CI or on a developer machine before cutting a release, never on the air-gapped target itself.</p>
<p><strong>Python wheelhouse:</strong></p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">./scripts/build_python_wheelhouse.sh                       # default: linux x86_64 + aarch64
./scripts/build_python_wheelhouse.sh --with-dev             # + requirements-dev.txt
./scripts/build_python_wheelhouse.sh --windows               # + win_amd64 wheels</code></pre></div>
<p>Downloads each dependency individually (<code>--no-deps</code>, then resolved separately) rather than as one batch, because pip&#39;s batch resolution is all-or-nothing — a single package needing a different manylinux tag can silently prevent the whole batch from downloading anything. Regenerates <code>vendor/python/CHECKSUMS.sha256</code> and verifies the result with a real <code>--no-index</code> install into a throwaway venv before finishing.</p>
<p><strong>Node/npm vendor artifacts:</strong></p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">./scripts/build_npm_offline_vendor.sh</code></pre></div>
<p>Runs <code>npm ci</code> for the local platform, then <code>npm pack</code> for each target platform&#39;s <code>@next/swc-*</code> native compiler package, tars the result into <code>vendor/npm/node_modules.tar.gz</code>, and writes checksums. See <code>vendor/npm/README.md</code> for the current state of what&#39;s vendored per platform — this is the one piece of the offline story where completing a full <code>node_modules.tar.gz</code> requires running the script end-to-end on a normal (non-sandboxed) machine, since it involves a full <code>npm ci</code> that can take longer than a constrained CI step allows. In practice this rarely matters day-to-day: <strong>the release bundle ships the pre-built <code>frontend/out/</code> static export</strong>, so an air-gapped install only needs the npm vendor artifacts if you&#39;re intentionally rebuilding the frontend from source offline.</p>
<h2 id="verifying-compliance-yourself">Verifying compliance yourself<a href="#verifying-compliance-yourself" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">python3 scripts/validate_airgap.py</code></pre></div>
<p>Prints a pass/fail line per check category and an overall summary. This is the same script CI runs on every push (<code>airgap-validation</code> job) — see <a href="../.github/workflows/ci.yml">.github/workflows/ci.yml</a>.</p>
<h2 id="why-this-doesnt-compromise-functionality">Why this doesn&#39;t compromise functionality<a href="#why-this-doesnt-compromise-functionality" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The one hard rule going into this work was: never remove a feature to make air-gap easier. Every existing capability — Swagger/ReDoc API docs, Excel import/export, MFA QR enrollment — still works exactly as before; the only change is that the assets those features depend on are now served from the same origin instead of a CDN.
