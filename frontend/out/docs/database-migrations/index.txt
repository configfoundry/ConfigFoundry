3:T4f3c,<h1 id="database-migrations">Database Migrations<a href="#database-migrations" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry v0.5 manages all database schema changes through <a href="https://alembic.sqlalchemy.org/" target="_blank" rel="noopener noreferrer">Alembic</a>, the standard SQLAlchemy migration toolkit. Every schema change is a versioned, reviewable Python file stored in <code>alembic/versions/</code>. Migrations are applied automatically at startup.</p>
<hr />
<h2 id="how-it-works">How it works<a href="#how-it-works" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">server.py                   configure_logging() + create_app()
    │
    ▼
ServiceContainer            builds SQLiteProvider (or future PostgreSQL/MySQL)
    │
    ▼
StorageProvider.initialize()
    │
    ▼
core.migrations.runner.run_migrations(engine)
    │
    ├── No alembic_version, no app tables  →  alembic upgrade head  (fresh DB)
    ├── No alembic_version, has app tables →  alembic stamp head    (legacy DB)
    └── Has alembic_version                →  alembic upgrade head  (no-op or pending)</code></pre></div>
<p><strong>Startup is always safe to run.</strong> If the database is already at head (the common case), <code>upgrade head</code> is a no-op. If new migrations are pending (after deploying a new version), they are applied automatically before any request is served.</p>

<h2 id="common-cli-commands">Common CLI commands<a href="#common-cli-commands" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Run from the project root. If the default SQLite URL in <code>alembic.ini</code> is not what you want, set <code>CONFIGFOUNDRY_DB_URL</code>:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">export CONFIGFOUNDRY_DB_URL=sqlite:///db/configfoundry.db
# or
export CONFIGFOUNDRY_DB_URL=postgresql+psycopg2://user:pass@host:5432/configfoundry</code></pre></div>
<h3 id="apply-all-pending-migrations">Apply all pending migrations<a href="#apply-all-pending-migrations" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic upgrade head</code></pre></div>
<p>Runs every migration that has not yet been applied to the target database. This is what the application does automatically on startup. Safe to run while the server is stopped.</p>
<h3 id="check-current-migration-state">Check current migration state<a href="#check-current-migration-state" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic current</code></pre></div>
<p>Shows which revision the database is currently at. Example output:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">c1f4e7a8b2d0 (head)</code></pre></div>
<h3 id="show-full-migration-history">Show full migration history<a href="#show-full-migration-history" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic history --indicate-current</code></pre></div>
<p>Lists all revisions in order with an arrow next to the current one:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">c1f4e7a8b2d0 -&gt; (head), Baseline schema — all tables at schema version 4

<h3 id="roll-back-one-migration">Roll back one migration<a href="#roll-back-one-migration" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic downgrade -1</code></pre></div>
<p>Rolls back the most recently applied migration. Use <code>alembic downgrade base</code> to roll back all migrations (drops all application tables).</p>
<h3 id="stamp-a-database-no-ddl">Stamp a database (no DDL)<a href="#stamp-a-database-no-ddl" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic stamp head</code></pre></div>
<p>Marks the database as being at the head revision without running any migrations. Used when taking over a database that was set up outside of Alembic. The runner does this automatically for databases created by the old custom migration system.</p>
<h3 id="preview-sql-offline-mode">Preview SQL (offline mode)<a href="#preview-sql-offline-mode" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic upgrade head --sql</code></pre></div>
<p>Generates the SQL DDL for pending migrations without executing it. Useful for DBA review, audit trails, or applying migrations through a controlled process.</p>
<hr />
<h2 id="adding-a-new-migration">Adding a new migration<a href="#adding-a-new-migration" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Every schema change — new table, new column, renamed column, added index — must be a new Alembic migration. Never modify an existing migration after it has been deployed.</p>
<h3 id="1-update-the-orm-model">1. Update the ORM model<a href="#1-update-the-orm-model" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>Edit the relevant model in <code>models/inventory.py</code>. For example, adding a <code>hostname</code> column to devices:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">class DeviceModel(Base):
    __tablename__ = &quot;devices&quot;

    id: Mapped[str] = mapped_column(String, primary_key=True)
    data: Mapped[str] = mapped_column(Text, nullable=False)
    updated_at: Mapped[float] = mapped_column(Float, nullable=False)
    hostname: Mapped[Optional[str]] = mapped_column(String, nullable=True)  # new</code></pre></div>
<h3 id="2-autogenerate-the-migration">2. Autogenerate the migration<a href="#2-autogenerate-the-migration" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>With the target database at head, run:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic revision --autogenerate -m &quot;add hostname column to devices&quot;</code></pre></div>
<p>Alembic compares the live database schema to the ORM models and generates a diff-based migration in <code>alembic/versions/</code>. Review the generated file carefully before committing — autogenerate is not perfect and may miss some differences (e.g. check constraints, some index types).</p>
<h3 id="3-review-the-generated-file">3. Review the generated file<a href="#3-review-the-generated-file" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>The generated file will look like:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># alembic/versions/a2b3c4d5e6f7_add_hostname_column_to_devices.py

revision = &quot;a2b3c4d5e6f7&quot;
down_revision = &quot;c1f4e7a8b2d0&quot;

def upgrade() -&gt; None:
    with op.batch_alter_table(&quot;devices&quot;) as batch_op:
        batch_op.add_column(sa.Column(&quot;hostname&quot;, sa.String(), nullable=True))

def downgrade() -&gt; None:
    with op.batch_alter_table(&quot;devices&quot;) as batch_op:
        batch_op.drop_column(&quot;hostname&quot;)</code></pre></div>
<p>Note the <code>op.batch_alter_table()</code> context — this is required for SQLite, which does not support <code>ALTER TABLE</code> directly. It is a no-op for PostgreSQL and MySQL.</p>
<h3 id="4-apply-and-test">4. Apply and test<a href="#4-apply-and-test" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">alembic upgrade head
python3 -m pytest tests/migrations/</code></pre></div>
<h3 id="5-commit">5. Commit<a href="#5-commit" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>Commit both the model change and the migration file together. The migration file is part of the application source code — it must be in version control.</p>
<hr />
<h2 id="sqlite-batch-mode">SQLite batch mode<a href="#sqlite-batch-mode" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>SQLite does not support most <code>ALTER TABLE</code> operations (add column to existing table, drop column, rename column). Alembic&#39;s <code>render_as_batch=True</code> option (enabled in <code>alembic/env.py</code>) wraps such operations in a table-copy-rename dance that achieves the same result.</p>
<p>This is transparent to callers. Use <code>op.batch_alter_table()</code> in all migrations that modify existing tables:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># Correct — works on SQLite, PostgreSQL, MySQL, SQL Server
def upgrade() -&gt; None:
    with op.batch_alter_table(&quot;devices&quot;) as batch_op:
        batch_op.add_column(sa.Column(&quot;hostname&quot;, sa.String(), nullable=True))

# Wrong — fails on SQLite
def upgrade() -&gt; None:
    op.add_column(&quot;devices&quot;, sa.Column(&quot;hostname&quot;, sa.String(), nullable=True))</code></pre></div>
<p><code>op.create_table()</code> and <code>op.drop_table()</code> (used in the baseline migration) do not require batch mode.</p>
<hr />
<h2 id="multi-database-support">Multi-database support<a href="#multi-database-support" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The migration files use SQLAlchemy column types (<code>sa.String</code>, <code>sa.Text</code>, <code>sa.Float</code>) rather than database-specific types, so they generate correct DDL for all supported backends:</p>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>SQLAlchemy type</th><th>SQLite</th><th>PostgreSQL</th><th>MySQL</th></tr></thead><tbody><tr><td><code>sa.String()</code></td><td>TEXT</td><td>VARCHAR</td><td>VARCHAR(255)</td></tr><tr><td><code>sa.Text()</code></td><td>TEXT</td><td>TEXT</td><td>TEXT</td></tr><tr><td><code>sa.Float()</code></td><td>REAL</td><td>DOUBLE PRECISION</td><td>DOUBLE</td></tr></tbody></table></div>
<p>Avoid database-specific SQL unless there is no cross-database alternative. If provider-specific SQL is unavoidable, gate it on the dialect:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">from alembic import op

def upgrade() -&gt; None:
    bind = op.get_bind()
    if bind.dialect.name == &quot;postgresql&quot;:
        # PostgreSQL-specific DDL
        op.execute(&quot;CREATE INDEX CONCURRENTLY ...&quot;)
    else:
        op.create_index(&quot;ix_devices_id&quot;, &quot;devices&quot;, [&quot;id&quot;])</code></pre></div>

<h2 id="existing-database-compatibility">Existing database compatibility<a href="#existing-database-compatibility" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<h3 id="databases-from-before-alembic-schema_version-4">Databases from before Alembic (schema_version 4)<a href="#databases-from-before-alembic-schema_version-4" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>The <code>core.migrations.runner</code> automatically detects databases that were created by the old <code>core/migrations_legacy.py</code> custom system. These have all application tables but no <code>alembic_version</code> table.</p>
<p>Detection: <code>meta</code> table exists AND <code>alembic_version</code> table does not.</p>
<p>Action: <code>alembic stamp head</code> — writes the baseline revision ID to <code>alembic_version</code> without touching any data or schema. On the next startup, the database is treated as Alembic-managed.</p>
<h3 id="databases-from-before-schema_version-4">Databases from before schema_version 4<a href="#databases-from-before-schema_version-4" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>If you have a database at schema_version 1–3, upgrade it to schema_version 4 first by running the old server one final time, then upgrade to the Alembic-managed version.</p>
<hr />
<h2 id="release-process">Release process<a href="#release-process" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>For every release that includes schema changes:</p>
<ol><li><strong>Ship the migration file</strong> in <code>alembic/versions/</code> alongside the application code.</li><li><strong>Stop the old server</strong> (or use a blue/green deployment).</li><li><strong>(Optional) Preview the SQL</strong>: <code>alembic upgrade head --sql &gt; migration.sql</code> — review before applying.</li><li><strong>Restart the new server</strong> — migrations run automatically on startup.</li></ol>
<p>The application aborts startup if a migration fails, leaving the database untouched.</p>
<hr />
<h2 id="rollback">Rollback<a href="#rollback" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>To roll back a migration after an incident:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash"># Check what&#39;s current
alembic current

# Roll back one migration
alembic downgrade -1

# Restart the old version of the application</code></pre></div>
<p>Not all schema changes are safely reversible. A migration that drops a column loses data — the downgrade would need to re-add the column, but the original data is gone. Plan downgrade paths carefully:</p>
<ul><li><strong>Additive changes</strong> (add column, add table, add index) — always safe to downgrade.</li><li><strong>Destructive changes</strong> (drop column, drop table) — reversible structurally but data is lost. Back up the database before applying.</li><li><strong>Data migrations</strong> (reshaping rows) — write explicit upward and downward data transforms in <code>upgrade()</code> and <code>downgrade()</code>.</li></ul>
<hr />
<h2 id="developer-guidelines">Developer guidelines<a href="#developer-guidelines" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong>Rule 1: Never edit a shipped migration.</strong> If a migration has been applied to any environment (dev, staging, production), it is immutable. Fix forward with a new migration.</p>
<p><strong>Rule 2: Migrations are code, not secrets.</strong> Commit migration files alongside the model changes that require them. A PR without the migration file is incomplete.</p>
<p><strong>Rule 3: Test migrations.</strong> Every migration should have a test in <code>tests/migrations/</code> that verifies the upgrade, checks the resulting schema, and (for reversible changes) verifies the downgrade.</p>
<p><strong>Rule 4: Use batch mode.</strong> Always use <code>op.batch_alter_table()</code> for modifications to existing tables so SQLite users are not left behind.</p>
<p><strong>Rule 5: Keep migrations provider-agnostic.</strong> Use SQLAlchemy column types. Document any unavoidable dialect-specific code with a comment explaining why.</p>
<p><strong>Rule 6: Autogenerate is a starting point, not the finish.</strong> Review the generated file. Alembic cannot detect: check constraints, renamed columns (it sees a drop + add), some custom types, server-side defaults on existing rows.</p>
