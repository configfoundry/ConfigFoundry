3:T4fc8,<h1 id="centralized-logging-framework">Centralized Logging Framework<a href="#centralized-logging-framework" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry v0.5 ships a self-contained logging framework in <code>core/logging/</code>. Every component of the application — FastAPI routes, services, repositories, middleware — logs through a single root logger named <code>configfoundry</code>, so all records can be captured, formatted, and rotated from one place.</p>
<hr />
<h2 id="at-a-glance">At a glance<a href="#at-a-glance" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">core/logging/
  __init__.py      # Public API: configure_logging(), get_logger(), ...
  config.py        # LoggingConfig dataclass (YAML / env / defaults)
  context.py       # ContextVar-based request ID propagation
  factory.py       # get_logger(__name__)  →  configfoundry.* namespace
  formatters.py    # ConfigFoundryFormatter (text) + JSONFormatter
  handlers.py      # build_console_handler(), build_file_handler()
  middleware.py    # CorrelationIDMiddleware + RequestLoggingMiddleware
  startup.py       # log_startup_info(), log_shutdown_info()</code></pre></div>

  provider: sqlite
  sqlite_path: db/configfoundry.db

  level: INFO                       # DEBUG | INFO | WARNING | ERROR | CRITICAL

<h3 id="environment-variables">Environment variables<a href="#environment-variables" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p>All logging options can also be set via <code>CONFIGFOUNDRY_LOG_*</code> variables, which take precedence over YAML defaults:</p>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Variable</th><th>Default</th><th>Description</th></tr></thead><tbody><tr><td><code>CONFIGFOUNDRY_LOG_LEVEL</code></td><td><code>INFO</code></td><td>Minimum log level (<code>DEBUG</code>, <code>INFO</code>, <code>WARNING</code>, <code>ERROR</code>, <code>CRITICAL</code>)</td></tr><tr><td><code>CONFIGFOUNDRY_LOG_FILE</code></td><td>—</td><td>Log file path; omit for console-only</td></tr><tr><td><code>CONFIGFOUNDRY_LOG_CONSOLE</code></td><td><code>true</code></td><td>Write to stderr (<code>true</code>/<code>false</code>)</td></tr><tr><td><code>CONFIGFOUNDRY_LOG_JSON</code></td><td><code>false</code></td><td>Emit JSON lines instead of text</td></tr><tr><td><code>CONFIGFOUNDRY_LOG_ROTATION</code></td><td><code>daily</code></td><td><code>daily</code>, <code>size</code>, or <code>none</code></td></tr><tr><td><code>CONFIGFOUNDRY_LOG_BACKUP_COUNT</code></td><td><code>7</code></td><td>Number of rotated files to keep</td></tr><tr><td><code>CONFIGFOUNDRY_LOG_MAX_BYTES</code></td><td><code>10485760</code></td><td>Max file size before rotation (<code>rotation=size</code>)</td></tr></tbody></table></div>

<h2 id="logger-hierarchy">Logger hierarchy<a href="#logger-hierarchy" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">configfoundry                       ← root (handlers attached here)
├── configfoundry.http              ← RequestLoggingMiddleware
├── configfoundry.lifecycle         ← startup / shutdown
├── configfoundry.core
│   ├── configfoundry.core.container
│   ├── configfoundry.core.storage.providers.sqlite
│   └── …
└── configfoundry.api
    └── configfoundry.api.v1.router
        └── …</code></pre></div>
<p>A single handler on the <code>configfoundry</code> root captures every record from any descendant. Setting <code>configfoundry.propagate = False</code> (done automatically by <code>configure_logging()</code>) ensures records never reach the stdlib root logger.</p>

<h2 id="using-the-logger-in-your-code">Using the logger in your code<a href="#using-the-logger-in-your-code" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">from core.logging import get_logger

logger = get_logger(__name__)   # always pass __name__

logger.debug(&quot;Checking device %s&quot;, device_id)
logger.info(&quot;Device created: %s&quot;, device_id)
logger.warning(&quot;Validation failed: %r&quot;, errors)
logger.error(&quot;Unexpected failure&quot;, exc_info=True)   # includes full traceback</code></pre></div>
<p><code>get_logger(__name__)</code> in <code>core/services/device_service.py</code> returns:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">logging.getLogger(&quot;configfoundry.core.services.device_service&quot;)</code></pre></div>
<p><strong>Never call <code>logging.getLogger()</code> directly.</strong> Always import <code>get_logger</code> from <code>core.logging</code>. This guarantees every logger lives in the <code>configfoundry</code> namespace.</p>
<hr />
<h2 id="correlation-ids">Correlation IDs<a href="#correlation-ids" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Every HTTP request is assigned a 12-character hex correlation ID (e.g. <code>a3f8c2d1e5b4</code>). The ID is stored in a <code>contextvars.ContextVar</code> so it propagates through the entire call stack — async handlers, sync handlers running in thread pools, and any intermediate awaits — without being passed as a function argument.</p>
<p>The ID is:</p>
<ul><li>Generated by <code>CorrelationIDMiddleware</code> if absent from the request.</li><li>Taken from the <code>X-Request-ID</code> request header if present (useful for tracing across service boundaries).</li><li>Added to every <code>X-Request-ID</code> response header so callers can correlate their own logs.</li><li>Injected into every log record automatically by <code>ConfigFoundryFormatter</code> and <code>JSONFormatter</code>.</li></ul>
<p>To read the current request ID in your code:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">from core.logging import get_request_id

def some_function():
    rid = get_request_id()   # &quot;-&quot; outside a request</code></pre></div>
<hr />
<h2 id="middleware-setup">Middleware setup<a href="#middleware-setup" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The middleware is registered in <code>app.py</code>. Starlette prepends middleware, so the last-registered runs first. The required registration order is:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># app.py — correct order
app.add_middleware(RequestLoggingMiddleware)   # added first → runs second
app.add_middleware(CorrelationIDMiddleware)    # added last  → runs first</code></pre></div>
<p><code>CorrelationIDMiddleware</code> must run first to set the <code>ContextVar</code> before <code>RequestLoggingMiddleware</code> logs the request. Reversing the order would log <code>[-]</code> instead of the real correlation ID.</p>
<h3 id="what-requestloggingmiddleware-logs">What RequestLoggingMiddleware logs<a href="#what-requestloggingmiddleware-logs" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">GET /api/v1/devices → 200 (12.3ms) ip=127.0.0.1</code></pre></div>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Field</th><th>Source</th></tr></thead><tbody><tr><td>Method</td><td><code>request.method</code></td></tr><tr><td>Path</td><td><code>request.url.path</code> (no query string)</td></tr><tr><td>Status</td><td><code>response.status_code</code></td></tr><tr><td>Duration</td><td><code>time.perf_counter()</code> delta</td></tr><tr><td>IP</td><td><code>X-Forwarded-For</code> header (first value) or <code>request.client.host</code></td></tr></tbody></table></div>
<p><strong>Never logged:</strong> request body, response body, query string parameters, authorization headers. If you need to log query parameters for debugging, do it inside the route handler with explicit redaction applied first.</p>
<hr />
<h2 id="exception-logging">Exception logging<a href="#exception-logging" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Unhandled exceptions are caught by the global exception handler in <code>app.py</code> and logged with full traceback — the client receives only a sanitized JSON response:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># What the client sees:
{&quot;error&quot;: &quot;division by zero&quot;, &quot;type&quot;: &quot;ZeroDivisionError&quot;}

# What the server logs (with exc_info=True):
2024-01-15 10:30:46 ERROR    configfoundry.app                  [a3f8c2d1e5b4] Unhandled ZeroDivisionError on GET /api/v1/devices
Traceback (most recent call last):
  ...
ZeroDivisionError: division by zero</code></pre></div>
<p>The correlation ID on the exception record lets you find the exact request in the access log.</p>

<hr />
<h2 id="log-rotation">Log rotation<a href="#log-rotation" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Mode</th><th>Handler</th><th>Behaviour</th></tr></thead><tbody><tr><td><code>daily</code></td><td><code>TimedRotatingFileHandler(when=&quot;midnight&quot;)</code></td><td>Rotate at midnight; keep <code>backup_count</code> files</td></tr><tr><td><code>size</code></td><td><code>RotatingFileHandler(maxBytes=max_bytes)</code></td><td>Rotate when file exceeds <code>max_bytes</code>; keep <code>backup_count</code></td></tr><tr><td><code>none</code></td><td><code>FileHandler</code></td><td>No rotation; use for development or short-lived jobs</td></tr></tbody></table></div>
<p>Parent directories are created automatically. There is no mode that allows unlimited log growth — the default <code>daily</code> rotation with <code>backup_count=7</code> caps total disk usage at roughly 7 × (daily write volume).</p>
<hr />
<h2 id="audit-log-future">Audit log (future)<a href="#audit-log-future" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The framework is designed for audit-log separation. When you&#39;re ready to add audit logging:</p>
<ol><li>Obtain the audit logger:</li></ol>
<p>   ``<code>python    audit = get_logger(&quot;configfoundry.audit&quot;)    </code>``</p>
<ol><li>Add a dedicated handler:</li></ol>
<p>   ``<code>python    import logging    audit_logger = logging.getLogger(&quot;configfoundry.audit&quot;)    audit_logger.addHandler(build_file_handler(&quot;logs/audit.log&quot;, rotation=&quot;daily&quot;))    audit_logger.propagate = False   # don&#39;t also write to the main log    </code>``</p>
<ol><li>The <code>request_id</code> ContextVar is automatically available — every audit record written during an HTTP request carries the correlation ID without any extra code.</li></ol>
<p>No code changes are needed in services or repositories; the infrastructure is already in place.</p>
<hr />
<h2 id="adding-a-new-log-destination">Adding a new log destination<a href="#adding-a-new-log-destination" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>To add a new handler type (e.g. a remote syslog server, Datadog agent, or email alerter on ERROR):</p>
<ol><li>Build the handler using stdlib <code>logging.handlers</code> or a third-party library.</li><li>Attach a <code>ConfigFoundryFormatter</code> or <code>JSONFormatter</code> so the <code>request_id</code> field is included.</li><li>Add it to <code>logging.getLogger(&quot;configfoundry&quot;)</code> either in <code>configure_logging()</code> (for always-on destinations) or at startup in <code>server.py</code> (for environment-specific sinks).</li></ol>
<hr />
<h2 id="testing-with-logging">Testing with logging<a href="#testing-with-logging" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>In tests, <code>configure_logging()</code> is not called (no <code>server.py</code> entrypoint runs). Loggers still work — they just have no handlers, so records are silently discarded. This is correct: tests should not write to disk or produce console noise unless you&#39;re specifically testing logging behaviour.</p>
<p>For tests that need to assert on log output, attach a capturing handler to the relevant logger:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python">import logging

class Capture(logging.Handler):
    def __init__(self):
        super().__init__()
        self.records = []
    def emit(self, record):
        self.records.append(record)

cap = Capture()
logging.getLogger(&quot;configfoundry&quot;).addHandler(cap)
# ... run the code under test ...
logging.getLogger(&quot;configfoundry&quot;).removeHandler(cap)

messages = [r.getMessage() for r in cap.records]
assert any(&quot;device created&quot; in m.lower() for m in messages)</code></pre></div>
<p>See <code>tests/logging/</code> for complete examples.</p>
