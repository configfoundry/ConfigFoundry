3:T3ac1,<h1 id="authentication-authorization--security">Authentication, Authorization &amp; Security<a href="#authentication-authorization--security" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry ships with a full authentication, RBAC, and Access Policy Engine layer. This document covers the architecture, configuration, and day-one operational steps (bootstrapping the first admin, issuing API keys, etc.). For the SOC 2 control mapping, see <a href="/docs/compliance-soc2/"><code>compliance-soc2.md</code></a>.</p>

                  │           │
                  │           └── refresh_tokens
                  │           └── mfa_backup_codes
                  │

                  └── network_acls</code></pre></div>
<p>All tables are defined in <code>models/auth.py</code>, created by <code>alembic/versions/0002_auth_and_security.py</code>. The existing <code>audit_log</code> table (<code>models/inventory.py</code>) was extended with nullable columns (<code>org_id</code>, <code>source_ip</code>, <code>user_agent</code>, <code>resource_type</code>, <code>resource_id</code>, <code>result</code>, <code>correlation_id</code>) rather than duplicated -- one audit trail for both business-object changes and security events.</p>
<hr />
<h2 id="authentication">Authentication<a href="#authentication" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<ul><li><strong>Password hashing</strong>: Argon2id (<code>core/security/password.py</code>), OWASP&#39;s</li></ul>
<p>  current recommendation. Parameters (time cost, memory cost,   parallelism) are configurable and embedded in the stored hash, so   raising them later doesn&#39;t invalidate existing hashes -- they&#39;re   upgraded transparently on next successful login (<code>needs_rehash</code>).</p>
<ul><li><strong>Password policy</strong>: NIST 800-63B-aligned -- length over complexity</li></ul>
<p>  theatre, no forced periodic rotation, adaptive hashing + account   lockout instead. Complexity toggles (mixed case / digit / symbol /   common-password rejection) exist and are on by default because some   compliance auditors expect to see them.</p>
<ul><li><strong>Tokens</strong>: short-lived JWT access tokens (15 min default,</li></ul>
<p>  <code>core/security/jwt_tokens.py</code>) + opaque, hashed, rotate-on-use refresh   tokens (14 days default) with reuse detection -- presenting an   already-rotated refresh token revokes its entire token family   immediately (OAuth 2.0 Security BCP response to suspected theft).</p>
<ul><li><strong>Session invalidation without a blacklist</strong>: every user row carries a</li></ul>
<p>  <code>perm_version</code> integer, embedded in each access token. Role changes,   forced logout, password changes, and account deactivation increment   it; <code>get_current_principal</code> rejects any token whose <code>perm_version</code>   doesn&#39;t match the current value. Effect: revocation is immediate, not   &quot;wait for the 15-minute token to expire,&quot; without needing a persistent   token blacklist.</p>
<ul><li><strong>Account lockout</strong>: configurable failed-attempt threshold (default 5)</li></ul>
<p>  locks the account for a configurable duration (default 15 min).</p>
<ul><li><strong>MFA</strong>: TOTP (RFC 6238, <code>core/security/mfa.py</code>), QR-code enrollment</li></ul>
<p>  with a two-step confirm flow, 10 one-time backup codes generated at   enrollment. MFA secrets are AES-256-GCM encrypted at rest   (<code>core/security/crypto.py</code>, via the audited <code>cryptography</code> package --   deliberately NOT <code>core/aesgcm.py</code>, which documents itself as   unsuitable for anything security-critical).</p>
<ul><li><strong>API keys</strong>: for service accounts / unattended scripts (e.g. SNMP</li></ul>
<p>  collectors calling the API without a human login). Hashed at rest,   shown in full exactly once at creation, optionally scoped by   permission list, IP allowlist, and expiration.</p>
<hr />
<h2 id="authorization-rbac">Authorization (RBAC)<a href="#authorization-rbac" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Permissions are <code>&lt;resource&gt;:&lt;action&gt;</code> codes (full catalog in <code>core/security/permissions.py</code>), never hardcoded role-name checks in route handlers -- every protected route depends on <code>require_permission(&quot;some:code&quot;)</code>. Five system roles are seeded with sensible default grants (Super Admin, Organization Admin, Operator, Read Only, Auditor); organizations may also define custom roles with any combination of permissions via <code>POST /api/v1/roles</code>.</p>
<p>Newly created users hold <strong>zero permissions</strong> until explicitly assigned a role -- there is no implicit access &quot;just because an account exists.&quot;</p>
<hr />
<h2 id="access-policy-engine">Access Policy Engine<a href="#access-policy-engine" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Runs before authentication (see pipeline above). IP allow/deny rules (<code>NetworkACL</code>, <code>core/services/policy_engine.py</code>) use first-match-wins evaluation ordered by <code>priority</code> (lower = evaluated first): the first rule whose CIDR contains the request IP decides the outcome. If nothing matches, the scope defaults to permissive UNLESS at least one <code>allow</code> rule exists for that scope, in which case presence of an allow rule signals &quot;allowlist mode&quot; and anything unmatched is denied by default. IPv4 and IPv6 both supported (stdlib <code>ipaddress</code>, no dependency).</p>
<p><strong>Not yet enforced</strong>: time-based access windows (maintenance windows / business hours) -- <code>PolicyEngine.evaluate_time_window()</code> exists as an extension point but always allows. Flagged here rather than silently absent.</p>
<hr />
<h2 id="configuration">Configuration<a href="#configuration" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>All settings are environment-variable driven (<code>CONFIGFOUNDRY_AUTH_*</code>), mirroring the existing <code>CONFIGFOUNDRY_DB_*</code> / <code>CONFIGFOUNDRY_LOG_*</code> convention. Full reference in <code>core/security/config.py</code>&#39;s module docstring. The most important ones for a production deployment:</p>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Variable</th><th>Purpose</th><th>Default</th></tr></thead><tbody><tr><td><code>CONFIGFOUNDRY_AUTH_JWT_SECRET</code></td><td>HMAC signing secret</td><td>randomly generated at startup (dev-only; <strong>set explicitly in production</strong> or every restart invalidates all sessions)</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_SECRET_ENC_KEY</code></td><td>base64, 32 bytes -- encrypts MFA seeds at rest</td><td>randomly generated at startup (same caveat)</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_BOOTSTRAP_EMAIL</code></td><td>first Super Admin&#39;s email</td><td><code>admin@configfoundry.local</code></td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_BOOTSTRAP_PASSWORD</code></td><td>first Super Admin&#39;s password</td><td>randomly generated, printed once to the server log at first migration</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_TRUSTED_PROXIES</code></td><td>comma-separated CIDRs allowed to set X-Forwarded-For</td><td>none</td></tr><tr><td><code>CONFIGFOUNDRY_AUTH_CORS_ORIGINS</code></td><td>comma-separated allowed origins</td><td>none (CORS closed by default)</td></tr></tbody></table></div>
<hr />
<h2 id="bootstrapping-the-first-admin">Bootstrapping the first admin<a href="#bootstrapping-the-first-admin" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>The auth migration (<code>0002_auth_and_security.py</code>) seeds one Super Admin user automatically. Set <code>CONFIGFOUNDRY_AUTH_BOOTSTRAP_EMAIL</code> and <code>CONFIGFOUNDRY_AUTH_BOOTSTRAP_PASSWORD</code> before the very first startup to control the credentials; otherwise a random password is generated and printed once to the console/log (never stored in plaintext, never a hardcoded <code>admin</code>/<code>admin</code> credential shipped in code) and the account is flagged <code>must_change_password</code>.</p>

<h3 id="interactive-docs-swagger-ui--redoc">Interactive docs (Swagger UI / ReDoc)<a href="#interactive-docs-swagger-ui--redoc" class="heading-anchor" aria-label="Link to this section">#</a></h3>
<p><code>/docs</code> and <code>/redoc</code> are registered as a <code>BearerAuth</code> HTTP security scheme in the OpenAPI document, so every protected route shows a lock icon and Swagger UI exposes an <strong>Authorize</strong> button. Paste either a JWT access token or an API key there (no <code>Bearer</code> prefix needed -- Swagger adds it) and &quot;Try it out&quot; will send it on every subsequent request from that tab.</p>
<hr />
<h2 id="known-scope-boundaries">Known scope boundaries<a href="#known-scope-boundaries" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Stated plainly so nothing here is a surprise later:</p>
<ul><li><strong>Multi-tenancy</strong>: the <code>Organization</code> model and the entire security</li></ul>
<p>  layer (users, roles, API keys, policies, audit) are fully org-scoped   and ready for multiple tenants. The pre-existing business tables   (devices, bandwidth, subnets, tags, lists, history) are <strong>not</strong> yet   retrofitted with <code>org_id</code> -- they remain single-tenant, scoped   implicitly to one seeded &quot;default&quot; organization. Retrofitting those   tables is a larger, separate migration touching all 8 existing   services and was out of scope for this pass.</p>
<ul><li><strong>External IdP / OIDC / SSO</strong>: not implemented. The token/claims model</li></ul>
<p>  is compatible with adding it later (an external identity would map to   the same <code>User</code> + <code>perm_version</code> + RBAC machinery) but no OIDC client   code exists yet.</p>
<ul><li><strong>Rate limiting</strong>: in-memory, per-process (<code>core/security/rate_limit.py</code>).</li></ul>
<p>  Fine for a single-process deployment; if ConfigFoundry is ever run as   multiple replicas without sticky sessions, each replica enforces its   own independent limit. Swapping in a Redis-backed limiter behind the   same interface would need no other code changes.</p>
<ul><li><strong>Time-based access policies</strong>: scaffolded, not enforced (see Access</li></ul>
<p>  Policy Engine section above).</p>
<ul><li><strong>GDPR-specific features</strong> (data export/erasure endpoints, consent</li></ul>
<p>  tracking): the audit trail and RBAC foundation support adding these,   but they are not built in this pass -- SOC 2 was the primary target.</p>
