3:T2afd,<h1 id="api-versioning-guide">API Versioning Guide<a href="#api-versioning-guide" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>ConfigFoundry v0.5 introduces URL-based API versioning. All REST endpoints are served under <code>/api/v1/</code>. Future versions (<code>/api/v2/</code>, <code>/api/v3/</code>) can be added alongside v1 without removing or changing any existing routes.</p>
<hr />
<h2 id="design-decisions">Design decisions<a href="#design-decisions" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong>URL versioning, not header versioning.</strong> The version appears in the path (<code>/api/v1/devices</code>) rather than a header (<code>Accept: application/vnd.api+json;version=1</code>). This makes versions:</p>
<ul><li>Bookmarkable and curl-able without extra flags</li><li>Visible in server logs without header inspection</li><li>Trivially testable with <code>status_code == 200</code> checks</li></ul>
<p><strong>Router-per-version, not branch-per-version.</strong> Each version is a self-contained Python package (<code>api/v1/</code>, <code>api/v2/</code>). Shared infrastructure (<code>api/dependencies.py</code>, <code>core/services/</code>, <code>core/repositories/</code>) is version-agnostic. A v2 router can re-use any v1 module it doesn&#39;t need to change.</p>
<p><strong>Single FastAPI app.</strong> All versions are included in one FastAPI application and appear together in the OpenAPI spec at <code>/docs</code>. This avoids running multiple processes and keeps the container wiring simple. If true isolation is needed (separate auth middleware, separate rate limits), the sub-application pattern described at the end of this document is an alternative.</p>

<h2 id="staticfiles-ordering-rule">StaticFiles ordering rule<a href="#staticfiles-ordering-rule" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><code>StaticFiles</code> is mounted at <code>/</code> in <code>create_app()</code> and acts as a catch-all. <strong>All <code>include_router()</code> calls must happen before <code>app.mount()</code>.</strong> Routers registered after the static mount are shadowed by it.</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># Correct order in app.py:
app.include_router(v1_router, prefix=&quot;/api&quot;)
app.include_router(v2_router, prefix=&quot;/api&quot;)   # add new versions HERE
# StaticFiles LAST:
app.mount(&quot;/&quot;, StaticFiles(directory=_static, html=True), name=&quot;static&quot;)</code></pre></div>

<h2 id="versioning-rules">Versioning rules<a href="#versioning-rules" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Rule</th><th>Rationale</th></tr></thead><tbody><tr><td>Never modify a shipped version&#39;s router</td><td>Clients depend on stable paths</td></tr><tr><td>Business logic lives in <code>core/services/</code>, not routers</td><td>Routers only translate HTTP ↔ service calls; services are version-agnostic</td></tr><tr><td>Schemas live in <code>schemas/</code></td><td>A v2 router can use new Pydantic models without touching v1 schemas</td></tr><tr><td>Keep the old version alive for ≥ one release cycle</td><td>Give clients time to migrate</td></tr><tr><td>Mark deprecated versions with a <code>Deprecated</code> response header</td><td>FastAPI lets you add custom middleware for this</td></tr></tbody></table></div>

<h2 id="alternative-sub-application-isolation">Alternative: sub-application isolation<a href="#alternative-sub-application-isolation" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>For strict version isolation (separate middleware stacks, separate auth, separate rate limits), each version can be a full <code>FastAPI</code> sub-application mounted with <code>app.mount()</code>:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">python</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-python"># app.py
from fastapi import FastAPI
from api.v1.router import router as v1_router

def create_v1_sub_app(container) -&gt; FastAPI:
    v1 = FastAPI(title=&quot;ConfigFoundry API v1&quot;, version=&quot;1.0.0&quot;)
    v1.state.container = container
    v1.include_router(v1_router)
    return v1

def create_app(...) -&gt; FastAPI:
    root = FastAPI()
    container = ServiceContainer(...)
    root.mount(&quot;/api/v1&quot;, create_v1_sub_app(container))
    # Docs at: /api/v1/docs
    return root</code></pre></div>
<p>Trade-offs vs. the single-app approach:</p>
<ul><li>✅ Each version has its own <code>/docs</code>, <code>/redoc</code>, <code>/openapi.json</code></li><li>✅ Middleware (auth, rate limit) can differ per version</li><li>✅ Can run different versions in different workers</li><li>❌ <code>app.state</code> is not shared — must pass <code>container</code> explicitly to each sub-app</li><li>❌ More boilerplate; <code>Depends(get_container)</code> must reference the sub-app&#39;s state</li></ul>
<p>The single-app approach (used by ConfigFoundry) is the right default. Switch to sub-apps when different versions genuinely need different middleware or deployment topology.</p>
