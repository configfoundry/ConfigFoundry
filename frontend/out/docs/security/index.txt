3:T1e41,<h1 id="security">Security<a href="#security" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>This page is the single entry point for ConfigFoundry&#39;s security model. It summarizes and cross-links the detail that lives in <a href="/docs/authentication/">Authentication</a>, <a href="/docs/authorization/">Authorization</a>, <a href="/docs/rbac/">RBAC</a>, and <a href="/docs/compliance-soc2/">compliance-soc2.md</a>, and adds the pieces that don&#39;t belong in any one of those: the CSP design, the vulnerability reporting process, and account-recovery procedures.</p>
<h2 id="threat-model-briefly">Threat model, briefly<a href="#threat-model-briefly" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>ConfigFoundry assumes it is deployed on a trusted internal network segment or behind a reverse proxy handling TLS, and that the operator controls who gets an account. It is not designed to be exposed directly to the public internet without a reverse proxy in front of it. Within that assumption, the security layer defends against: credential theft (short-lived tokens, MFA, Argon2id hashing), privilege escalation (explicit permission grants, no implicit access), session hijacking (refresh token rotation with reuse detection), and unauthorized network access (the IP-based Access Policy Engine).</p>
<h2 id="authentication-summary">Authentication summary<a href="#authentication-summary" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Argon2id password hashing, short-lived JWT access tokens (15 min default) with rotate-on-use refresh tokens (14 days default) that detect reuse and revoke the entire token family if an already-rotated token is presented, TOTP-based MFA with encrypted-at-rest seeds, and API keys for service accounts. Full detail: <a href="/docs/authentication/">Authentication</a>.</p>
<h2 id="authorization-summary">Authorization summary<a href="#authorization-summary" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Every protected route depends on a permission code (<code>require_permission(&quot;resource:action&quot;)</code>), never a hardcoded role name. New accounts start with zero permissions. Role changes take effect immediately via a <code>perm_version</code> check, without needing a token blacklist. Full detail: <a href="/docs/authorization/">Authorization</a>, full permission catalog: <a href="/docs/rbac/">RBAC</a>.</p>
<h2 id="content-security-policy">Content-Security-Policy<a href="#content-security-policy" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="code-block"><div class="code-block-header"><span class="code-lang">text</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-">default-src &#39;self&#39;; script-src &#39;self&#39; &#39;unsafe-inline&#39;; style-src &#39;self&#39; &#39;unsafe-inline&#39;;
img-src &#39;self&#39; data:; connect-src &#39;self&#39;; frame-ancestors &#39;none&#39;</code></pre></div>
<p>Applied uniformly to every response by <code>SecurityHeadersMiddleware</code> (<code>core/security/middleware.py</code>). Same-origin only — no CDN allowlisting anywhere, which the air-gap requirement forced and which happens to also be the stricter, more defensible policy: everything the frontend and the <code>/docs</code>/<code>/redoc</code> pages need is self-hosted (see <a href="/docs/airgap/">Air-Gap Deployment</a>).</p>
<p>The one relaxation is <code>&#39;unsafe-inline&#39;</code> on <code>script-src</code>, required because Next.js&#39;s App Router embeds its hydration payload in an inline <code>&lt;script&gt;</code> tag — this is a framework constraint, not a choice made for convenience, and it does not weaken the policy against loading <em>external</em> script sources, which remains fully locked to <code>&#39;self&#39;</code>.</p>
<h2 id="other-security-headers">Other security headers<a href="#other-security-headers" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Alongside CSP, every response gets <code>Strict-Transport-Security</code>, <code>X-Frame-Options: DENY</code> (belt-and-suspenders alongside <code>frame-ancestors &#39;none&#39;</code>), and standard MIME-sniffing protection. HSTS is inert over plain HTTP and takes effect once TLS is terminated at your reverse proxy — see <a href="/docs/deployment/">Deployment</a>.</p>
<h2 id="rate-limiting">Rate limiting<a href="#rate-limiting" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Per-IP, in-memory, stricter on <code>/auth/login</code> and <code>/auth/mfa/*</code> (<code>CONFIGFOUNDRY_AUTH_RATE_LIMIT_LOGIN</code>, default <code>10/60</code>). Known limitation: this is per-process, so a multi-instance deployment enforces independent limits per instance rather than one shared global limit — see <a href="/docs/authentication/#known-scope-boundaries">Authentication § Known scope boundaries</a>.</p>
<h2 id="audit-trail">Audit trail<a href="#audit-trail" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Every security-relevant event (login, MFA change, role change, API key lifecycle, policy change) and every business-data mutation is recorded via <code>AuditRepository.log(...)</code>, queryable at <code>GET /api/v1/audit/search</code> (requires <code>audit:read</code>). Nothing is auto-purged — plan storage growth accordingly if retention matters for your compliance requirements.</p>
<h2 id="recovering-from-a-lost-admin-account">Recovering from a lost admin account<a href="#recovering-from-a-lost-admin-account" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>If the bootstrap Super Admin&#39;s credentials are lost and no other admin account exists:</p>
<ol><li>If you have direct database access, the cleanest path is a small,</li></ol>
<p>   deliberate script using the same <code>core/security/password.py</code> Argon2id    hasher to set a new password hash directly on the user row — do this    once, out-of-band, not as a supported CLI command (there isn&#39;t one,    by design, to avoid shipping a built-in backdoor).</p>
<ol><li>Alternatively, delete the database and let ConfigFoundry re-bootstrap</li></ol>
<p>   on next startup — only viable if you&#39;re comfortable losing existing    inventory/security data, so this is a last resort, not a first one.</p>
<ol><li>For a production deployment, avoid this scenario entirely: always</li></ol>
<p>   have at least two Super Admin or Organization Admin accounts.</p>
<h2 id="known-scope-boundaries">Known scope boundaries<a href="#known-scope-boundaries" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Restated here from <a href="/docs/authentication/">Authentication</a> since they&#39;re security-relevant: no external IdP/OIDC/SSO integration yet; rate limiting is per-process, not distributed; time-based (business-hours) access policies are scaffolded but not enforced; multi-tenancy covers the security layer but not yet the inventory tables. None of these are silently absent — each is called out explicitly so a deployer can decide whether it matters for their environment.</p>
<h2 id="reporting-a-vulnerability">Reporting a vulnerability<a href="#reporting-a-vulnerability" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>If you find a security issue, report it privately rather than opening a public GitHub issue — use GitHub&#39;s private security advisory feature on the repository, or contact the maintainer directly. Include the affected version, reproduction steps, and impact assessment if you have one. There is no bug bounty program at this time.</p>5:["slug","security","d"]
