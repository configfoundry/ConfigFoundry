3:T1b77,<h1 id="troubleshooting">Troubleshooting<a href="#troubleshooting" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<h2 id="startup">Startup<a href="#startup" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong>&quot;python3 not found&quot; / &quot;Python 3.10+ required&quot;</strong> Install Python 3.10 or newer and ensure it&#39;s on <code>PATH</code>. Check with <code>python3 --version</code>. <code>install_offline.sh</code> fails fast with this exact message rather than a stack trace further in.</p>
<p><strong>Server starts but the browser tab shows a blank page</strong> This was a real bug class during development, root-caused to the Content-Security-Policy being stricter than what the frontend actually needs. If you&#39;re running a build predating that fix, or you&#39;ve customized the CSP in <code>core/security/middleware.py</code>, check that <code>script-src</code> includes <code>&#39;unsafe-inline&#39;</code> — Next.js&#39;s App Router embeds an inline hydration <code>&lt;script&gt;</code> tag that a stricter policy silently blocks with no visible error except a blank page. See <a href="/docs/security/#content-security-policy">Security § Content-Security-Policy</a>.</p>
<p><strong><code>/docs</code> or <code>/redoc</code> won&#39;t load, or loads with missing styling</strong> Both pages are fully self-hosted (<code>static/vendor/docs/</code>) specifically so this doesn&#39;t depend on CDN access. If they&#39;re broken, check that <code>static/vendor/docs/swagger-ui-bundle.js</code>, <code>swagger-ui.css</code>, and <code>redoc.standalone.js</code> exist and aren&#39;t truncated (<code>ls -la static/vendor/docs/</code> — compare sizes against a known-good release). A truncated download during a build interruption is the most likely cause if this ever regresses.</p>
<p><strong>Bootstrap admin password wasn&#39;t printed / I lost it</strong> It&#39;s printed exactly once, to the console/log, at first migration — never stored in plaintext anywhere and not recoverable. If you missed it and the account still has <code>must_change_password</code> set with no known password, use another Super Admin account to call <code>POST /api/v1/users/{id}/reset-password</code>, or, if this is truly the only admin account and you have direct database access, see <a href="/docs/security/">Security § Recovering from a lost admin account</a>.</p>
<h2 id="offline-install">Offline install<a href="#offline-install" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong><code>install_offline.sh</code> fails at &quot;Offline install failed&quot;</strong> Usually means <code>vendor/python/</code> doesn&#39;t contain a wheel matching this exact machine&#39;s platform/Python version tags. Run <code>python3 -m pip debug --verbose</code> to see this machine&#39;s compatible tags and compare against <code>vendor/python/CHECKSUMS.sha256</code>. See <a href="/docs/airgap/#targeting-a-different-platform">Air-Gap Deployment § Targeting a different platform</a>.</p>
<p><strong><code>install_offline.sh</code> says Node.js not found and no <code>frontend/out/</code></strong> The release bundle should always ship a pre-built <code>frontend/out/</code>, making Node unnecessary. If you&#39;re working from a source checkout instead of a release bundle and <code>frontend/out/</code> genuinely isn&#39;t there, either install Node 18+ and let the script attempt an offline rebuild from <code>vendor/npm/</code>, or copy a pre-built <code>frontend/out/</code> over from a machine that has one.</p>
<p><strong><code>scripts/validate_airgap.py</code> fails on &quot;external URL not in allowlist&quot;</strong> A new file added a <code>http://</code>/<code>https://</code> reference that isn&#39;t in <code>ALLOWED_URL_SUBSTRINGS</code>. If it&#39;s a genuine CDN/remote-asset dependency, vendor it instead (see <a href="/docs/airgap/">Air-Gap Deployment</a>). If it&#39;s a harmless reference that&#39;s never actually fetched (a doc link, an XML namespace URI), add it to the allowlist in <code>scripts/validate_airgap.py</code> with a comment explaining why it&#39;s safe.</p>
<h2 id="authentication">Authentication<a href="#authentication" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong>Logged in but every request returns 403</strong> The account has zero permissions — new users start with none until a role is explicitly assigned (<code>POST /api/v1/users/{id}/roles</code>). This is deliberate, not a bug — see <a href="/docs/rbac/">RBAC</a>.</p>
<p><strong>Getting logged out immediately after a role change</strong> Expected behavior — <code>perm_version</code> is bumped on role changes, which invalidates existing access tokens immediately rather than waiting for natural expiry. Log in again to get a token reflecting the new permissions. See <a href="/docs/authorization/">Authorization § Immediate revocation</a>.</p>
<p><strong>Locked out after failed login attempts</strong> Wait out <code>CONFIGFOUNDRY_AUTH_LOCKOUT_MINUTES</code> (default 15), or have another admin reset the account. Threshold and duration are both configurable — see <a href="/docs/configuration/">Configuration</a>.</p>
<p><strong>Every restart logs everyone out</strong> <code>CONFIGFOUNDRY_AUTH_JWT_SECRET</code> isn&#39;t set, so a new random one is generated on every startup, invalidating all previously issued tokens. Set it explicitly — see <a href="/docs/configuration/#authentication--security-configfoundry_auth_">Configuration § Set this in production</a>.</p>
<h2 id="database">Database<a href="#database" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p><strong>&quot;database is locked&quot; errors under SQLite</strong> SQLite allows only one writer at a time. This surfaces under concurrent write-heavy load from multiple processes/instances pointed at the same file. Either reduce write concurrency, or move to PostgreSQL — see <a href="/docs/storage/">Storage</a>.</p>
<p><strong>Migration didn&#39;t apply / app won&#39;t start after upgrade</strong> Migrations run automatically at startup and should never require a manual step. If startup fails with a migration error, check <code>alembic current</code> against the expected head for the version you&#39;re upgrading to, and see <a href="/docs/migrations/">Migrations</a> and <a href="/docs/upgrade/">Upgrade Guide</a> for the safe rollback path (restore the pre-upgrade backup <code>upgrade_offline.sh</code> creates automatically).</p>
