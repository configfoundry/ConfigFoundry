3:T1b8c,<h1 id="role-based-access-control-rbac">Role-Based Access Control (RBAC)<a href="#role-based-access-control-rbac" class="heading-anchor" aria-label="Link to this section">#</a></h1>
<p>The full permission catalog and system role definitions, as seeded by the auth migration (<code>alembic/versions/0002_auth_and_security.py</code>) and kept self-consistent by <code>core/security/permissions.py::ensure_seeded()</code> (runs at every <code>ServiceContainer</code> startup, so a newly added permission code in a future release self-heals into the database without a manual migration step).</p>
<p>For how these are enforced at the route level, see <a href="/docs/authorization/">Authorization</a>. For how a user gets a role assigned, see the <code>/users/{id}/roles</code> endpoints in <a href="/docs/api/">API Reference</a>.</p>
<h2 id="permission-catalog">Permission catalog<a href="#permission-catalog" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Code</th><th>Category</th><th>Grants</th></tr></thead><tbody><tr><td><code>inventory:read</code></td><td>Inventory</td><td>View devices, bandwidth caps, subnets, tags, and lists</td></tr><tr><td><code>inventory:write</code></td><td>Inventory</td><td>Create, edit, delete, and import inventory records</td></tr><tr><td><code>profile:read</code></td><td>Deployment</td><td>View generated YAML config output and history</td></tr><tr><td><code>profile:write</code></td><td>Deployment</td><td>Modify config-generation inputs</td></tr><tr><td><code>deployment:execute</code></td><td>Deployment</td><td>Run YAML config generation</td></tr><tr><td><code>export:read</code></td><td>Deployment</td><td>Export inventory data to Excel</td></tr><tr><td><code>settings:modify</code></td><td>System</td><td>Change application-wide settings</td></tr><tr><td><code>meta:read</code></td><td>System</td><td>View inventory metadata and statistics</td></tr><tr><td><code>user:read</code></td><td>Access Management</td><td>View user accounts</td></tr><tr><td><code>user:manage</code></td><td>Access Management</td><td>Create, edit, deactivate, and assign roles to users</td></tr><tr><td><code>role:read</code></td><td>Access Management</td><td>View roles and their permissions</td></tr><tr><td><code>role:manage</code></td><td>Access Management</td><td>Create, edit, and delete roles; change permission grants</td></tr><tr><td><code>api:manage</code></td><td>Access Management</td><td>Create, rotate, and revoke API keys</td></tr><tr><td><code>policy:manage</code></td><td>Access Management</td><td>Manage IP allow/deny lists and access policies</td></tr><tr><td><code>organization:manage</code></td><td>Access Management</td><td>Manage organization settings (multi-tenant)</td></tr><tr><td><code>audit:read</code></td><td>Audit</td><td>View the audit log (business + security events)</td></tr></tbody></table></div>
<h2 id="system-roles">System roles<a href="#system-roles" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Five roles are seeded automatically. All except Super Admin can be freely edited — reassign their permissions, or create entirely new custom roles — through <code>POST /api/v1/roles</code> and <code>PATCH /api/v1/roles/{id}/permissions</code> (requires <code>role:manage</code>).</p>
<div class="table-wrap"><table class="docs-table"><thead><tr><th>Role</th><th>Intended for</th><th>Permissions</th></tr></thead><tbody><tr><td><strong>Super Admin</strong></td><td>Platform operators</td><td>Every permission, always in sync with the catalog. Not org-scoped — spans every organization. Not editable through the UI (holding everything is definitional).</td></tr><tr><td><strong>Organization Admin</strong></td><td>Team/department leads</td><td>Full access within one organization: <code>inventory:*</code>, <code>profile:*</code>, <code>deployment:execute</code>, <code>export:read</code>, <code>settings:modify</code>, <code>meta:read</code>, <code>user:*</code>, <code>role:*</code>, <code>api:manage</code>, <code>policy:manage</code>, <code>audit:read</code></td></tr><tr><td><strong>Operator</strong></td><td>Day-to-day inventory work</td><td><code>inventory:read</code>, <code>inventory:write</code>, <code>profile:read</code>, <code>profile:write</code>, <code>deployment:execute</code>, <code>export:read</code>, <code>meta:read</code> — no user/role/policy management</td></tr><tr><td><strong>Read Only</strong></td><td>Stakeholders who need visibility, not edit access</td><td><code>inventory:read</code>, <code>profile:read</code>, <code>export:read</code>, <code>meta:read</code></td></tr><tr><td><strong>Auditor</strong></td><td>Compliance/security review</td><td><code>inventory:read</code>, <code>profile:read</code>, <code>meta:read</code>, <code>audit:read</code> — read-only plus the audit trail, deliberately without operational access</td></tr></tbody></table></div>
<h2 id="custom-roles">Custom roles<a href="#custom-roles" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Any combination of the permission codes above can be assembled into a custom role via <code>POST /api/v1/roles</code>:</p>
<div class="code-block"><div class="code-block-header"><span class="code-lang">bash</span><button type="button" class="copy-btn" aria-label="Copy code">Copy</button></div><pre><code class="language-bash">curl -sX POST http://localhost:8420/api/v1/roles \
  -H &quot;Authorization: Bearer $TOKEN&quot; -H &#39;Content-Type: application/json&#39; \
  -d &#39;{
    &quot;name&quot;: &quot;Bandwidth Reviewer&quot;,
    &quot;description&quot;: &quot;Read inventory, review generated configs, no write access&quot;,
    &quot;permissions&quot;: [&quot;inventory:read&quot;, &quot;profile:read&quot;, &quot;meta:read&quot;]
  }&#39;</code></pre></div>
<p>A user can hold multiple roles simultaneously — their effective permission set is the union of every assigned role&#39;s grants.</p>
<h2 id="design-rationale">Design rationale<a href="#design-rationale" class="heading-anchor" aria-label="Link to this section">#</a></h2>
<p>Permission codes rather than role-name checks means the <em>set of permissions</em> is the stable contract routes depend on, while <em>which roles grant which permissions</em> stays fully data-driven and editable without a code change or redeploy. Adding a new permission is:</p>
<ol><li>Add an entry to <code>PERMISSION_CATALOG</code> in <code>core/security/permissions.py</code>.</li><li><code>ensure_seeded()</code> picks it up automatically on next startup.</li><li>Reference it from a route with <code>require_permission(&quot;new:code&quot;)</code>.</li><li>Optionally add it to <code>SYSTEM_ROLES</code> defaults, or leave it for admins</li></ol>
<p>   to grant to custom roles as needed.</p>
